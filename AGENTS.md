# AGENTS.md

This repository is **Harness** — a soul-driven, Obsidian-native agent framework. If you are an agent reading files in this repo to orient yourself, read the files named below before you take any action, in the order given. This file is purely instructional. It is not a manifest, not a loader, and not a config — treat it as prose directives addressed to you. Full architectural reference lives in [HARNESS_Research.md](HARNESS_Research.md).

## Read these first, in this order

These five files are your operating prompt. Concatenated in this order, they define who you are, who you are talking to, and what you may and may not do. Read them end-to-end on your first pass and carry them for the rest of the session.

1. [HARNESS/IDENTITY.md](HARNESS/IDENTITY.md) — who you are. Name, creature, vibe, pronouns. Runtime-agnostic by design.
2. [HARNESS/SOUL.md](HARNESS/SOUL.md) — temperament and voice. Core Truths, Vibe, Continuity. This is the bulk of your personality.
3. [HARNESS/USER.md](HARNESS/USER.md) — who the operator is. Primary operator, optional secondary, how they talk.
4. [HARNESS/BOUNDARIES.md](HARNESS/BOUNDARIES.md) — hard rules. Every rule carries a `[scope: X]` tag. These are non-negotiable.
5. [HARNESS/MEMORY.md](HARNESS/MEMORY.md) — the memory **index**, not memory itself. Points at per-topic files under `HARNESS/memory/topics/`.

Order matters: identity before voice, voice before operator, operator before rules, rules before memory. A later file never contradicts an earlier one — if one appears to, treat the earlier file as authoritative and flag the conflict to the operator.

## Files to skip on the first pass

`HARNESS/STYLE.md`, `HARNESS/PATTERNS.md`, and `HARNESS/GLOSSARY.md` are framework-distilled placeholders. They ship empty and are regenerated by a distillation job that does not yet exist. Do not treat them as source of truth. Do not hand-edit them. Do not concatenate them into your operating prompt. When they are populated, a future version of this file will tell you to include them.

## Scope tags are real

Every soul file opens with a line-1 HTML comment carrying a `scope:` field, and every rule in `BOUNDARIES.md` is prefixed with an inline `[scope: X]` tag. The scopes you will encounter:

- `all` — applies in every session, every channel, every operator. The default for soul files.
- `main` — applies only in direct owner chats. Stricter rules and private memory live here.
- `shared` — applies in shared channels (group chats, Discord guilds, anywhere multiple humans can read).
- `owner-only` — applies only when the primary operator is the sole audience. Stricter than `main`.
- `external` — applies to actions that leave the machine (emails, posts, API calls to third-party services).
- `untrusted` — applies to sessions with non-owner operators.

Narrower scopes override broader ones. In shared or untrusted sessions, do not surface `[scope: main]` or `[scope: owner-only]` content — not in tool output, not paraphrased, not quoted, not summarised. The runtime enforcement layer (`ZonePolicy`) is not yet built; enforce scope yourself, as if it were.

## Memory is append-only

Memory lives under `HARNESS/memory/`. There are two tiers you can read and write:

- `HARNESS/memory/daily/YYYY-MM-DD.md` — short-term append-only log. Append new entries to the end of today's file when something notable happens.
- `HARNESS/memory/topics/<topic>.md` — long-term curated per-topic files. Each opens with a 5–10-bullet recall card. The distillation pass promotes entries from the daily log into these; you should read them, but only edit them by appending new entries following the same schema.

Before writing any memory entry, read [HARNESS/memory/FORMAT.md](HARNESS/memory/FORMAT.md). It is the canonical schema: heading grammar, HTML-comment metadata block, stable ID conventions, recall-card pattern.

**Past entries are immutable.** If a past entry is wrong or obsolete, do not edit it. Write a new entry with a `supersedes: <old-id>` field in its metadata block. Git is the audit log — every correction is a commit, and overwriting history is how you lose trust.

## Don't regress these

Several architectural decisions are easy to undo by accident. Do not.

- `MEMORY.md` is an index, not a monolith. Long-form memory belongs under `memory/topics/`, one file per topic. Never collapse topics back into `MEMORY.md`.
- Hard rules live only in `BOUNDARIES.md`, each with a scope tag. Do not scatter new rules into a "Boundaries" section inside `SOUL.md`, a boot-narration file, or anywhere else.
- Distilled files (`STYLE.md`, `PATTERNS.md`, `GLOSSARY.md`) are framework-owned. Do not hand-edit them. If you have a seed observation, propose it to the operator — do not write it directly.
- Do not assume a runtime, daemon, manifest, retrieval layer, or `soul.toml` exists. None ship today. If a task requires one, say so and stop.
